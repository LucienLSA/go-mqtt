# go-mqtt 项目全景图与架构说明

## 1. 项目定位
本项目是一个后端优先的物联网设备管理平台，围绕设备接入、鉴权、状态同步、数据采集、控制下发和回执闭环构建。

当前已具备后端 MVP 闭环能力：
- 设备管理（CRUD）
- 管理端鉴权（JWT）与权限控制（RBAC）
- EMQX 接入鉴权（HTTP Auth）
- EMQX 事件回调（Webhook）状态回写
- MQTT 上报数据订阅与落库
- 控制下发、回执处理、命令历史
- 命令超时与重试机制

## 2. 项目全景图（逻辑视角）

### 2.1 参与方
- 管理端调用方（通过 HTTP API 管理设备）
- 设备或网关（通过 MQTT 接入 Broker）
- EMQX Broker（处理 MQTT，回调后端鉴权与事件）
- 后端服务（Gin + GORM + MQTT Subscriber）
- MySQL（持久化设备、数据、命令、用户）

### 2.2 主链路
1. 管理端登录获取 JWT。
2. 管理端通过带 Token 的接口进行设备管理与控制下发。
3. EMQX 在设备连接时调用后端 Auth 接口判定 allow/deny。
4. 设备上报数据到 MQTT Topic，后端订阅并落库。
5. 控制命令发布到设备控制 Topic，设备反馈回执。
6. 后端根据 trace_id 更新命令状态；超时命令进入重试或终止。

## 4. 关键知识点（按能力域）

### 4.1 双认证体系分离
- 管理端 API: JWT + RBAC（人访问平台）
- 设备接入: EMQX HTTP Auth（设备/网关接入 Broker）

这种分离是 IoT 平台常见架构，避免把管理权限与设备接入权限混用。

### 4.2 Neuron 与设备账号认证并行
- 网关认证路径：NEURON_GATEWAY_USERNAME / NEURON_GATEWAY_PASSWORD
- 设备认证路径：username=device_id，password=device_secret

网关模式便于统一接入，设备模式便于细粒度控制与审计。

### 4.3 Webhook 事件边界
只对核心事件进入状态回写逻辑：
- client.connected
- client.disconnected

其他事件统一忽略，减少误触发与业务噪音。

### 4.4 Webhook 安全基线
已实现：
- 来源 IP 白名单
- HMAC 签名头校验

价值：防伪造请求、防越权回调、提升接入安全性。

### 4.5 命令可靠性状态机
命令不是一次发布就结束，而是可观测、可重试的流程：
- pending
- retrying
- success / timeout(failed)

核心字段：retry_count、max_retry、timeout_at、next_retry_at、done_at。

### 4.6 分层设计思想
- Handler 处理协议与参数。
- Repository 处理数据库。
- MQTT 模块处理消息与异步任务。
- Middleware 处理通用安全能力。

该分层有利于后续扩展测试、监控和微服务拆分。

## 9. 面试高频问题与回答要点

### 9.1 为什么要双认证体系？
- 管理员和设备是两类主体，安全模型不同。
- JWT + RBAC 适合人机接口与细粒度功能授权。
- EMQX HTTP Auth 适合连接时快速鉴权与接入控制。

### 9.2 为什么要做命令重试，不直接失败？
- IoT 场景常见弱网、离线和瞬时抖动。
- 单次失败会造成业务误判，重试可提高最终成功率。
- 结合超时上限和最大重试次数，避免无限重试带来的系统压力。

### 9.3 trace_id 的价值是什么？
- 把“下发请求、Broker 发布、设备回执、命令状态”关联成一条链路。
- 支持问题排查、审计和可观测性建设。

### 9.4 Webhook 为什么要做 HMAC + 白名单？
- 白名单防止非预期来源访问。
- HMAC 保证消息来源可信且内容未被篡改。

## 15. 端到端控制闭环说明（文字 + 代码定位）

本章节用于面试讲解，目标是说明一条控制命令如何从 HTTP 请求进入系统，经 MQTT 到达设备，再通过回执回写数据库并被历史接口查询到。

### 15.1 全链路一句话
脚本先登录并创建设备，然后启动模拟器，调用控制接口下发命令；后端生成 trace_id 并写入命令日志后发布 MQTT 控制消息；模拟器回发带 trace_id 的反馈；后端消费反馈并幂等更新命令状态；脚本最后按 trace_id 查询历史并判断 status 是否为 1。

### 15.2 分阶段说明

阶段 A：登录与创建设备
- 脚本调用 `/api/v1/auth/login` 获取 JWT。
- 使用 JWT 调用 `/api/v1/device` 创建设备，得到 `id` 和 `device_id`。
- 这里的 `id` 用于 HTTP 控制接口，`device_id` 用于 MQTT topic。

阶段 B：启动模拟器
- 脚本后台执行 `go run ./cmd/simulator -device-id <device_id>`。
- 模拟器连接 Broker 后会：
  - 周期发布 `device/<device_id>/data` 上报消息。
  - 订阅 `device/<device_id>/control` 控制主题。

阶段 C：下发控制命令
- 脚本调用 `POST /api/v1/device/:id/control`。
- 控制接口先经过限流与熔断保护。
- Handler 中会：
  - 生成 `trace_id`。
  - 先插入 `command_log`（状态 `pending`，即 `status=0`）。
  - 再发布 MQTT 控制消息到 `device/<device_id>/control`。
- 接口响应返回 `trace_id`，用于后续链路匹配。

阶段 D：回执消费与幂等更新
- 模拟器收到控制消息后，发布反馈到 `device/<device_id>/feedback`。
- 后端订阅器消费反馈并解析 `trace_id/result/message`。
- 数据库更新条件为 `trace_id=? AND status=0`：
  - 首次回执可更新成功（`RowsAffected > 0`）。
  - 重复回执更新 0 行，被识别为重复并忽略（幂等）。

阶段 E：历史查询判定
- 脚本等待几秒后调用 `GET /api/v1/device/:id/command?limit=20`。
- 在历史列表里按 `trace_id` 精确匹配本次命令。
- 判定规则：
  - `status=1` 视为成功。
  - 未匹配或非 1 视为失败并输出详情，便于排障。

- 脚本流程：`scripts/mock/e2e_control_flow.sh`
- 模拟器收发：`cmd/simulator/simulator.go`
- 控制接口处理：`internal/handler/device_handler.go`
