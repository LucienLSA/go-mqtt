# go-mqtt 项目全景图与架构说明

## 1. 项目定位
本项目是一个后端优先的物联网设备管理平台，围绕设备接入、鉴权、状态同步、数据采集、控制下发和回执闭环构建。

当前已具备后端 MVP 闭环能力：
- 设备管理（CRUD）
- 管理端鉴权（JWT）与权限控制（RBAC）
- EMQX 接入鉴权（HTTP Auth）
- EMQX 事件回调（Webhook）状态回写
- MQTT 上报数据订阅与落库
- 控制下发、回执处理、命令历史
- 命令超时与重试机制

## 2. 项目全景图（逻辑视角）

### 2.1 参与方
- 管理端调用方（通过 HTTP API 管理设备）
- 设备或网关（通过 MQTT 接入 Broker）
- EMQX Broker（处理 MQTT，回调后端鉴权与事件）
- 后端服务（Gin + GORM + MQTT Subscriber）
- MySQL（持久化设备、数据、命令、用户）

### 2.2 主链路
1. 管理端登录获取 JWT。
2. 管理端通过带 Token 的接口进行设备管理与控制下发。
3. EMQX 在设备连接时调用后端 Auth 接口判定 allow/deny。
4. 设备上报数据到 MQTT Topic，后端订阅并落库。
5. 控制命令发布到设备控制 Topic，设备反馈回执。
6. 后端根据 trace_id 更新命令状态；超时命令进入重试或终止。

- cmd: 入口程序（server/simulator/deploy）
- internal: 核心业务代码（handler、mqtt、repository、model 等）
- scripts/mock: Linux 模拟测试脚本
- openapi.json: API 文档
- .env: 环境变量配置

### 3.2 启动入口层
- cmd/server/main.go
  - 服务启动、加载 .env、初始化 DB、启动 MQTT 订阅器、注册路由。
- cmd/simulator/
  - 设备模拟器，支持定时上报与控制回执。

### 3.3 Handler 层
- internal/handler/auth_handler.go
  - 登录与当前用户信息。
- internal/handler/device_handler.go
  - 设备管理、控制下发、命令历史查询。
- internal/handler/emqx_handler.go
  - EMQX HTTP Auth 与 Webhook 处理。
  - 已加入来源 IP 白名单与 HMAC 签名校验。
  - 核心事件边界：client.connected、client.disconnected。

### 3.4 Middleware 与 Auth
- internal/middleware/auth.go
  - JWT 鉴权中间件与角色权限校验。
- internal/auth/jwt.go
  - JWT 生成与解析。
- internal/auth/password.go
  - bcrypt 哈希与密码校验。
- internal/auth/user_store.go
  - 启动种子用户配置解析。

### 3.5 MQTT 与异步处理
- internal/mqtt/subscriber.go
  - 订阅上报主题与反馈主题。
  - 发布控制消息。
  - 回执处理与命令状态更新。

  - Device、SensorData、CommandLog（含 retry/timeout 字段）。

  - sensor_data_repository.go

### 3.7 基础设施

  - DB 连接与 AutoMigrate。

### 3.8 测试与模拟

## 4. 关键知识点（按能力域）

### 4.1 双认证体系分离
- 管理端 API: JWT + RBAC（人访问平台）
- 设备接入: EMQX HTTP Auth（设备/网关接入 Broker）

这种分离是 IoT 平台常见架构，避免把管理权限与设备接入权限混用。

### 4.2 Neuron 与设备账号认证并行
- 网关认证路径：NEURON_GATEWAY_USERNAME / NEURON_GATEWAY_PASSWORD
- 设备认证路径：username=device_id，password=device_secret

网关模式便于统一接入，设备模式便于细粒度控制与审计。

### 4.3 Webhook 事件边界
只对核心事件进入状态回写逻辑：
- client.connected
- client.disconnected

其他事件统一忽略，减少误触发与业务噪音。

### 4.4 Webhook 安全基线
已实现：
- 来源 IP 白名单
- HMAC 签名头校验

价值：防伪造请求、防越权回调、提升接入安全性。

### 4.5 命令可靠性状态机
命令不是一次发布就结束，而是可观测、可重试的流程：
- pending
- retrying
- success / timeout(failed)

核心字段：retry_count、max_retry、timeout_at、next_retry_at、done_at。

### 4.6 分层设计思想
- Handler 处理协议与参数。
- Repository 处理数据库。
- MQTT 模块处理消息与异步任务。
- Middleware 处理通用安全能力。

该分层有利于后续扩展测试、监控和微服务拆分。

## 6. 推荐学习与实践路线
1. 先跑通全链路：登录 -> 创建设备 -> 模拟上报 -> 下发控制 -> 查看命令状态。
2. 再观察可靠性：停掉模拟器制造超时，检查重试与终态。
3. 最后压测与安全验证：批量设备、Webhook 签名、IP 白名单、权限边界。

通过这三步，你可以完整掌握 IoT 平台后端从功能可用到工程可用的核心能力。
